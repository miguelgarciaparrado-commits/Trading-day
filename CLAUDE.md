## CRITICAL: Build Pipeline

**Vercel/Netlify serves static HTML тАФ there is NO build step on the hosting platform.**

Every code change requires:
```bash
node build.js               # JSX тЖТ trading-diary.html
cp trading-diary.html index.html  # index.html is what the root URL serves
git add trading-diary.jsx trading-diary.html index.html
git commit -m "..."
git push -u origin <branch>
```

**Never** push only the `.jsx` file тАФ the deployed app will not change.

### How `build.js` works

1. Reads `trading-diary.jsx`, strips `import`/`export` statements
2. Compiles with TypeScript API: `jsx: React, target: ES2020, module: None`
3. Post-processes: removes `"use strict"`, `exports.*`, `Object.defineProperty(exports...)` artifacts
4. Removes orphan commas (TypeScript bug: treats `<` `>` in JSX as generics, inserts trailing commas)
5. Prepends `const {useState,useEffect,useRef,useCallback} = React;`
6. Splices compiled JS into `trading-diary.html` between two fixed markers:
   - Start: `const {useState,useEffect,useRef,useCallback} = React;`
   - End: `var root = ReactDOM.createRoot(document.getElementById("root"));`

`node --check trading-diary.html` does **not** work (Node.js rejects `.html` extension). To verify JS syntax, extract the script block manually.

### JSX Constraints (TypeScript/UMD React, no Babel)

- No `?.` optional chaining тАФ use `(x && x.prop) || default`
- No `??` nullish coalescing тАФ use `|| default`
- No `async ({destructured}) =>` in parameters тАФ use `async function(param){ const {a,b}=param; }`
- No IIFEs `(()=>{...})()` inside JSX тАФ extract to named function
- No template literal types or TypeScript generics
- All JSX must have balanced divs per conditional branch

## Architecture

### Stack
- **React 18 UMD** (loaded from CDN, no bundler)
- **CSS-in-JS** тАФ all styles are inline objects in `const S = {...}` at module level
- **Binance REST + WebSocket** тАФ price data and real-time klines
- **Supabase** тАФ primary persistence (PostgreSQL, single row per user)
- **localStorage** тАФ offline backup, always written synchronously before Supabase
- **Anthropic API** (Claude) тАФ chat tab + profile analysis generation

### Data Persistence

```
Edit тЖТ localStorage (immediate, _savedAt timestamp) тЖТ Supabase (600ms debounce)
Load тЖТ fetch BOTH Supabase + localStorage тЖТ use whichever has newer _savedAt
```

Every write to `td-user` (localStorage or Supabase payload) includes `_savedAt: Date.now()`.
On load, both sources are fetched and compared by `_savedAt`; the fresher one wins.
This prevents Supabase (slow async write) from overwriting a more recent localStorage save on page reload.

Main data blob saved under key `td-user` / Supabase `trading_data`:
```javascript
{
  pr:       { BTC: 84000, SOL: 91, ETH: 2000, ... },  // last known prices
  pos:      [ ...open positions ],
  pats:     [ ...chart pattern book ],
  jnl:      [ ...psychological diary entries ],
  ps:       { slOk, slBroken, slBreakeven, earlyClose, tpAuto, tpManual, revenge,
               manualClose, ratioSum, ratioCount, tpStreak, bestTpStreak },
  xhist:    [ ...closed positions via app ],
  ethClosed: false,
  predictions: [...],
  chatMsgs: [...],
  _savedAt: 1713000000000   // timestamp ms тАФ used to resolve Supabase vs localStorage conflicts
}
```

### Critical: Immediate localStorage Writes

`closePos`, `autoCheckPositions`, ETH close, and manual trade add ALL write to localStorage **synchronously** (before the debounced Supabase call). This prevents data loss if the page is closed within the 600ms debounce window.

Pattern used everywhere:
```javascript
try{localStorage.setItem("td-user",JSON.stringify({...payload,_savedAt:Date.now()}));}catch(e){}
save(); // debounced Supabase sync
```

## Position System

### Position Object Schema
```javascript
{
  id: number,
  asset: string,          // e.g. "BTC", "SOL"
  dir: "Long" | "Short",
  capital: number,        // USD capital
  entry: number,          // entry price
  sl: number,             // stop loss price
  tp: number,             // single take profit (OR use tpLevels)
  tpLevels: [             // partial takes (ventas parciales)
    { id, price, pct, hit }  // pct = % of position to close at this level
  ],
  capitalRemaining: number, // after partial takes
  be: boolean,            // breakeven flag
  date: string,
  note: string
}
```

### Direction Validation Rules
- **LONG**: SL < entry, TP > entry
- **SHORT**: SL > entry, TP < entry
- Always use `Math.abs()` for dollar amounts; validate sides separately

### Auto-Close Logic (`autoCheckPositions`)
Runs after every `fetchPrices()` cycle (~60s). Checks all open positions:
- SL hit тЖТ closes position, saves to `xhist` with "ЁЯЫС SL auto" or "тЪЦя╕П BE auto" note
- TP level hit тЖТ closes that % of position, marks level `hit:true`
- All TP levels hit тЖТ removes position entirely, one consolidated xhist entry
- Sends Telegram + browser notification via `notifyAutoClose()`
- Saves immediately via `saveRef.current()` (no debounce) + synchronous localStorage write

### Ratio R:R Tracking
When a position closes (any method), `ratio` is saved in the `xhist` entry and accumulated in `ps.ratioSum` / `ps.ratioCount`. The Resumen tab shows "RATIO R:R MEDIO". **SL hits do NOT count toward ratio average тАФ only TPs.**

### Manual Trade Add (Historial tab)
"+ A├▒adir trade" button in Historial tab. Validates all fields and shows `alert()` on missing data (no silent failures). If close price is left empty, defaults to entry price (= breakeven). Writes to localStorage immediately on add.

## Alert System (AlertasTab)

### How Alerts Work
Each alert = one Binance WebSocket kline stream (`symbol@kline_interval`).
Historical candles pre-loaded via REST on start (250 candles).
OHLC stored in `ohlcRef.current[symbol+interval]`.
RSI history stored in `rsiHistRef.current[symbol+interval]`.
EMA state stored in `emaStateRef.current`.

### Automatic Triggers (no user toggle required)
- RSI enters тЙд30 zone тЖТ `"rsi_oversold"` (fires once on zone entry, **2-point hysteresis**: resets only when RSI > 32)
- RSI enters тЙе70 zone тЖТ `"rsi_overbought"` (resets only when RSI < 68)
- EMA 7 crosses above EMA 25 тЖТ `"golden"`
- EMA 7 crosses below EMA 25 тЖТ `"death"`
- EMA 50 crosses above EMA 200 тЖТ `"ema200_golden"`
- EMA 50 crosses below EMA 200 тЖТ `"ema200_death"`
- RSI divergence on closed candle тЖТ `"rsi_div_bull"` or `"rsi_div_bear"` (time-based cooldown = 1 candle period)

**RSI and divergence are fully independent signals** тАФ RSI notifications do NOT show divergence data.

### Optional User-Configured Triggers
- Custom RSI level (`rsiCustomEnabled`, `rsiCustomTarget`, `rsiCustomCondition`)
- FVG covered (`fvgEnabled`)

### Channel Detection (`detectChannelAlert`)
Detects ascending and descending parallel channels on closed candles only (`k.x === true`).

**Algorithm:**
1. Takes last 80 candles, detects pivot highs/lows with `detectPivots(highs, lows, 2)` (window=2 тЖТ 4-candle confirmation)
2. `detectPivots` returns `{pH:[{x,y}], pL:[{x,y}]}` тАФ **always use `.x`/`.y` accessors, never `[0]`/`[1]`**
3. Runs `linReg` on pivot arrays тЖТ upper and lower trendlines
4. Parallelism check: same slope direction + ratio 0.20тАУ5.0
5. Touch validation: pivot within 35% of channel height from its line counts as a touch (min 1 each side)
6. Breakout: price > topLine├Ч1.002 (up) or price < botLine├Ч0.998 (down)
7. False breakout: needs BOTH long wick AND close still inside channel
8. Zone alerts: `pos < 0.18` тЖТ support, `pos > 0.82` тЖТ resistance, middle тЖТ no alert

**State machine (WS handler):**
- New channel type detected (including first detection from `prevCT=null`): fires immediately unless localStorage cooldown active (`td-chan-{ak}{canalType}`, cooldown = 3├Ч interval period)
- Same channel, zone transitions: supportтЖФresistance fires on zone change; breakout fires once per channel (`chan_hp_done` flag)
- Re-entry after breakout: fires re-entry alert
- Channel lost (`detectChannelAlert` returns null): resets all state

**`startAlert` initialises per alert:** `chan_type=null`, `chan_zone=null`, `chan_was_out=false`, `chan_hp_done=false`

### Cooldown / Anti-spam
`lastTrigRef.current[key]` tracks fired state per alert+type. Fires once when condition becomes true, resets when it becomes false.
- RSI zones: 2-point hysteresis buffer (overbought resets at <68, oversold resets at >32)
- Divergence: time-based lock = one full candle period
- Channels: localStorage timestamp, cooldown = 3├Ч interval period (survives WS reconnect)

### Alert Object Schema
```javascript
{
  id: number,
  symbol: string,           // e.g. "BTCUSDT"
  label: string,            // display name
  interval: "1h"|"4h"|"1d"|"1w",
  rsiCustomEnabled: boolean,
  rsiCustomTarget: number,  // 31-69
  rsiCustomCondition: "above"|"below",
  fvgEnabled: boolean,
  active: boolean,
  currentRsi: number|null,
  currentPrice: number|null,
  error: boolean
}
```

### Notifications
Every `sendAlert()` call:
1. Logs to in-app `logs` state (shown in Alertas tab)
2. Sets `lastAlert` state (banner shown for 8s)
3. Vibrates device (if supported)
4. Sends Telegram message (if token + chat ID configured)
5. Shows browser Notification (if permission granted)

## Auditor├нa de operaciones тАФ compatibilidad con registros hist├│ricos

El bot├│n "Auditar" eval├║a operaciones cerradas contra 5 reglas de disciplina
(R1тАУR5, ver `AUDIT_SYSTEM_PROMPT`). Las operaciones cerradas antes de que
existiera la feature de auditor├нa no tienen todos los campos enriquecidos
(`sl_initial`, `sl_modifications`, `thesis_text`, etc.). El auditor es
**tolerante**: audita con lo que hay y marca expl├нcitamente lo que no puede.

### Clasificaci├│n de campos

```javascript
AUDIT_CORE_FIELDS     = ["asset","dir","result","date","note"];
AUDIT_ENRICHED_FIELDS = ["sl_initial","sl_modifications","thesis_text",
                         "ratio","patternId","broker","reopen_after_sl"];
```

- **Core:** identidad de la operaci├│n. Si falta algo de esto тЖТ la operaci├│n
  se marca `nonAuditable` (sin llamada a la API).
- **Enriched:** necesarios para puntuar reglas espec├нficas. Si faltan тЖТ se
  informa al modelo en el prompt para que devuelva `status:"na"` en esas
  reglas, no `"fail"`.

### Helpers (top-level en `trading-diary.jsx`)

| Funci├│n | Rol |
|---|---|
| `detectMissingFields(op)` | `{core, enriched, hasMinimum}` |
| `hasMinimumAuditData(op)` | atajo booleano |
| `buildNonAuditableReport(op, missing)` | reporte sin AI para ops sin core |

### Flujo de `runAuditForEntry(entryId)`

1. Buscar `trade` en `D.current.xhist`. Si no existe тЖТ `console.warn` y salir.
2. Calcular `missing = detectMissingFields(trade)`; `console.warn` si faltan.
3. Si `!missing.hasMinimum` тЖТ persistir `nonAuditable` y volver.
4. Llamar `auditTrade(trade, null, missing)`; el prompt incluye los campos
   faltantes y pide marcarlos como `"na"`.
5. Sea cual sea el resultado (├йxito, error API, excepci├│n), **siempre** se
   persiste un `audit_report`. Nunca se falla en silencio.

### Estados UI en el modal

| Estado | Trigger | Render |
|---|---|---|
| тЬЕ pass | regla cumplida | badge verde |
| тЭМ fail | regla rota | badge rojo |
| ЁЯФШ N/D | `status==="na"` | badge gris + explicaci├│n |
| Auditor├нa limitada | `missingEnriched.length тЙе 3` | badge `ЁЯУЛ` en listado + nota en modal |
| No auditable | `nonAuditable:true` | panel ├бmbar con raz├│n + campos disponibles |
| Error API | `audit_report.error` | panel rojo + lista de campos faltantes + bot├│n reaudit |

### Restricciones

- **No migrar datos hist├│ricos.** Las operaciones viejas conservan su
  estructura original. No se rellenan campos sint├йticamente.
- **No cambiar la l├│gica de las 5 reglas.** El auditor ya usa `"na"` para
  evaluaciones sin datos тАФ solo hay que avisarle de qu├й falta.
- La racha `ps.tpStreak` y los KPIs (`avgScore`, `audited`, `pending`) filtran
  `nonAuditable` para no contaminar estad├нsticas.

## Invariantes de direcci├│n en alertas FVG

Contratos que CUALQUIER cambio al m├│dulo de imbalances debe respetar. Auditado
en PR #23, tests en `src/imbalances/fvg-direction-tests.js`.

### I1 тАФ Clasificaci├│n de direcci├│n
Los detectores (`checkFVGCovered`, `detectVolumeImbalance`, `detectOpeningGap`,
`detectLiquidityVoid`) DEBEN devolver objetos con:
- `subtype: "alcista" | "bajista"` тАФ direcci├│n del imbalance
- `kind: "fvg" | "vi" | "gap" | "void"` тАФ tipo
- `top`, `bot` тАФ bordes num├йricos (top > bot siempre)

### I2 тАФ Entrada en borde proximal
- **Long (FVG alcista) con precio arriba del FVG:** entry = `top` (primer borde
  que el precio toca al bajar).
- **Short (FVG bajista) con precio abajo del FVG:** entry = `bot` (primer borde
  que el precio toca al subir).

Implementaci├│n correcta: `sendAlert` bloque `patron_fvg` l├нnea ~4823
(`fvgEdge = fvgIsLong ? fvgR.top : fvgR.bot`).

### I3 тАФ SL en el lado contrario + colch├│n ATR
- **Long:** SL `< bot` del FVG, restando `0.3 ├Ч ATR(14)` de la TF correspondiente.
- **Short:** SL `> top` del FVG, sumando `0.3 ├Ч ATR(14)`.

Nunca SL del mismo lado que la direcci├│n del trade.

### I4 тАФ TP en la direcci├│n del trade
- **Long:** el siguiente imbalance usado como TP DEBE ser `subtype === "alcista"`
  y su precio `> entry`.
- **Short:** el siguiente imbalance usado como TP DEBE ser `subtype === "bajista"`
  y su precio `< entry`.

Prohibido usar un imbalance de direcci├│n contraria como TP (romper├нa la l├│gica
operativa тАФ el precio tendr├нa que romper resistencia/soporte para llegar).

### I5 тАФ `findFVGforTP` filtra por direcci├│n + posici├│n
La funci├│n acepta `bullish: boolean` y filtra candidatos por:
1. Direcci├│n del imbalance (alcista si `bullish`, bajista si no).
2. Posici├│n relativa a `refPrice` (por encima si bullish, por debajo si no).

`findNearFVG` filtra SOLO por direcci├│n y distancia тАФ no por posici├│n arriba/abajo
del `refPrice`. Usarla solo para detectar "d├│nde estamos parados", nunca para TP.

### I6 тАФ R/R positivo y coherente (тЙе 1.0)
- `R/R = |TP тИТ entry| / |entry тИТ SL|` usando `Math.abs()` тЖТ siempre positivo.
- Si un cap de TP (por ejemplo `tpMaxPct` por TF en l├нnea ~4661) recorta el TP
  de tal manera que `R/R < 1.0`, la se├▒al DEBE marcarse "R/R insuficiente" o
  bloquearse тАФ nunca emitirse como OPERACI├УN v├бlida.

## Cascada de TP por TF en multi-TF OPERACI├УN

**Regla:** la TF **menor** de la confluencia dicta el rango v├бlido de TP.

| TF menor de la confluencia | Rango TP v├бlido |
|---|---|
| 1H | 0.8% тАУ 2.5% |
| 4H | 2.0% тАУ 5.0% |
| 1D | 4.0% тАУ 10.0% |
| 1W | 8.0% тАУ 25.0% |

**Ejemplo:** confluencia 1H + 1D тЖТ `lowestTf = "1h"` тЖТ TP debe estar en **0.8тАУ2.5%** del entry.
Un FVG de 1D a +6% se rechaza aunque est├й dentro del rango 1D, porque la TF menor (1H) manda.

### Implementaci├│n

Variable `lowestTf` calculada justo despu├йs de `tfConfs` (bloque multi-TF en `sendAlert`):
```javascript
var lowestTf = interval; // fallback
for (var i = 0; i < ["1h","4h","1d","1w"].length; i++) {
  if (tfConfs.some(c => c.tf === TF_LIST_ORD[i])) { lowestTf = TF_LIST_ORD[i]; break; }
}
```

`tpMaxPct` y `tpMinPct` se derivan de `TP_RANGES[lowestTf]` (no de `interval`).

Cascada de b├║squeda de TP:
1. FVG pr├│ximo en `ohlcData` (TF de la se├▒al) dentro del rango `[tpMinPct, tpMaxPct]` con R/R тЙе 1.5.
2. FVG lejano multi-TF (`fvgFarExit`) dentro del mismo rango.
3. Se├▒ales espec├нficas de tipo (EMA50, canal opuesto, soporte/resistencia estructura).
4. **Swing estructural** de `ohlcByTf[lowestTf]` тАФ ├║ltimas 30 velas, nivel H/L m├бs cercano en rango.
5. Si ninguno aplica тЖТ `tpManualTarget = true` тЖТ se emite la se├▒al **sin TP** con nota `"gesti├│n manual del objetivo"`.

El cap final tambi├йn respeta `tpMaxPct` derivado de `lowestTf`.

### Nota en Telegram

Cuando `lowestTf !== interval` (la TF menor no es la del alert), la l├нnea de TP incluye:
```
ЁЯОп TP: $X (+1.8%) тАФ FVG 1H [rango dictado por 1H: 0.8тАУ2.5%]
```
Cuando no hay TP v├бlido:
```

```

## PWA + Web Push Notifications

The app is an installable PWA on Android with native push notifications fanned
out by a Supabase Edge Function (`send-push`). Telegram runs in parallel as a
fallback during the transition window.

### Files

| File | Role |
|------|------|
| `manifest.json` | PWA manifest (install metadata, icons, shortcuts) |
| `sw.js` | Service Worker (push handler, notificationclick router, cache shell) |
| `icon.svg` | App icon тАФ referenced by manifest and notifications |
| `migrations/add_push_subscriptions.sql` | `public.push_subscriptions` DDL (one row per device) |
| `supabase/functions/send-push/index.ts` | Edge Function: fan-out push to active subs via `web-push` |
| `scripts/generate-vapid.js` | Generate VAPID keypair (Node built-in crypto, no deps) |

### Setup (one-time)

```bash
# 1. Generate VAPID keys
node scripts/generate-vapid.js
# тЖТ copy VAPID_PUBLIC_KEY into trading-diary.jsx (constant VAPID_PUBLIC_KEY)
# тЖТ store both keys as Supabase secrets

# 2. Create the DB table
# Open Supabase SQL editor тЖТ paste migrations/add_push_subscriptions.sql тЖТ run

# 3. Set Edge Function secrets
supabase secrets set VAPID_PUBLIC_KEY=<public>
supabase secrets set VAPID_PRIVATE_KEY=<private>
supabase secrets set VAPID_SUBJECT=mailto:you@example.com

# 4. Deploy the Edge Function
supabase functions deploy send-push --no-verify-jwt

# 5. Rebuild and redeploy the frontend
node build.js && cp trading-diary.html index.html
git add тАж && git commit -m "тАж" && git push
```

### Subscription flow

1. User opens Alertas tab тЖТ clicks ЁЯФФ button.
2. `subscribePush()` requests `Notification.permission` (Chrome Android shows OS prompt).
3. On grant: `pushManager.subscribe({userVisibleOnly:true, applicationServerKey:VAPID_PUBLIC_KEY})`.
4. Endpoint + `p256dh` + `auth` are POSTed to `push_subscriptions` via PostgREST.
5. Button turns green; the SW will now handle incoming push events even when the tab is closed.

Unsubscribe: pulsar el bot├│n ЁЯФФ activo тЖТ `sub.unsubscribe()` + PATCH row `is_active=false`.

### Deep link routing

The SW's `notificationclick` handler opens/focuses a client window at the
`deepLink` URL. `trading-diary.jsx` reads `?tab=N` on load and sets the active
tab. When `action=open_trade`, a flag is persisted to
`localStorage["td-pending-trade-action"]` so the Posiciones tab can pick it up.

Tab indices (see `TABS` array):

| Tab | Index | URL |
|---|---|---|
| Resumen | 0 | `/?tab=0` |
| Posiciones | 1 | `/?tab=1` |
| Historial | 2 | `/?tab=2` |
| Alertas | 7 | `/?tab=7` |
| Chat | 8 | `/?tab=8` |

### Fallback behaviour

- Telegram still fires on every `sendAlert` / `notifyAutoClose` тЖТ push is purely additive right now.
- If the push endpoint returns 404/410 (expired subscription), the Edge Function marks it `is_active=false` automatically.
- `pushsubscriptionchange` event in the SW posts a message to open clients so the app can re-subscribe (automatic renewal: future enhancement).

### iOS note

Out of scope for this PR. iOS Safari requires the user to install the PWA to
the home screen **before** push works, and even then the VAPID delivery path
has quirks. Android Chrome is the primary target.
