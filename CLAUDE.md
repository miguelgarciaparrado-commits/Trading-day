## CRITICAL: Build Pipeline

**Vercel/Netlify serves static HTML вҖ” there is NO build step on the hosting platform.**

Every code change requires:
```bash
node build.js               # JSX вҶ’ trading-diary.html
cp trading-diary.html index.html  # index.html is what the root URL serves
git add trading-diary.jsx trading-diary.html index.html
git commit -m "..."
git push -u origin <branch>
```

**Never** push only the `.jsx` file вҖ” the deployed app will not change.

### How `build.js` works

1. Reads `trading-diary.jsx`, strips `import`/`export` statements
2. Compiles with TypeScript API: `jsx: React, target: ES2020, module: None`
3. Post-processes: removes `"use strict"`, `exports.*`, `Object.defineProperty(exports...)` artifacts
4. Removes orphan commas (TypeScript bug: treats `<` `>` in JSX as generics, inserts trailing commas)
5. Prepends `const {useState,useEffect,useRef,useCallback} = React;`
6. Splices compiled JS into `trading-diary.html` between two fixed markers:
   - Start: `const {useState,useEffect,useRef,useCallback} = React;`
   - End: `var root = ReactDOM.createRoot(document.getElementById("root"));`

`node --check trading-diary.html` does **not** work (Node.js rejects `.html` extension). To verify JS syntax, extract the script block manually.

### JSX Constraints (TypeScript/UMD React, no Babel)

- No `?.` optional chaining вҖ” use `(x && x.prop) || default`
- No `??` nullish coalescing вҖ” use `|| default`
- No `async ({destructured}) =>` in parameters вҖ” use `async function(param){ const {a,b}=param; }`
- No IIFEs `(()=>{...})()` inside JSX вҖ” extract to named function
- No template literal types or TypeScript generics
- All JSX must have balanced divs per conditional branch

## Architecture

### Stack
- **React 18 UMD** (loaded from CDN, no bundler)
- **CSS-in-JS** вҖ” all styles are inline objects in `const S = {...}` at module level
- **Binance REST + WebSocket** вҖ” price data and real-time klines
- **Supabase** вҖ” primary persistence (PostgreSQL, single row per user)
- **localStorage** вҖ” offline backup, always written synchronously before Supabase
- **Anthropic API** (Claude) вҖ” chat tab + profile analysis generation

### Data Persistence

```
Edit вҶ’ localStorage (immediate, _savedAt timestamp) вҶ’ Supabase (600ms debounce)
Load вҶ’ fetch BOTH Supabase + localStorage вҶ’ use whichever has newer _savedAt
```

Every write to `td-user` (localStorage or Supabase payload) includes `_savedAt: Date.now()`.
On load, both sources are fetched and compared by `_savedAt`; the fresher one wins.
This prevents Supabase (slow async write) from overwriting a more recent localStorage save on page reload.

Main data blob saved under key `td-user` / Supabase `trading_data`:
```javascript
{
  pr:       { BTC: 84000, SOL: 91, ETH: 2000, ... },  // last known prices
  pos:      [ ...open positions ],
  pats:     [ ...chart pattern book ],
  jnl:      [ ...psychological diary entries ],
  ps:       { slOk, slBroken, slBreakeven, earlyClose, tpAuto, tpManual, revenge,
               manualClose, ratioSum, ratioCount, tpStreak, bestTpStreak },
  xhist:    [ ...closed positions via app ],
  ethClosed: false,
  predictions: [...],
  chatMsgs: [...],
  _savedAt: 1713000000000   // timestamp ms вҖ” used to resolve Supabase vs localStorage conflicts
}
```

### Critical: Immediate localStorage Writes

`closePos`, `autoCheckPositions`, ETH close, and manual trade add ALL write to localStorage **synchronously** (before the debounced Supabase call). This prevents data loss if the page is closed within the 600ms debounce window.

Pattern used everywhere:
```javascript
try{localStorage.setItem("td-user",JSON.stringify({...payload,_savedAt:Date.now()}));}catch(e){}
save(); // debounced Supabase sync
```

## Position System

### Position Object Schema
```javascript
{
  id: number,
  asset: string,          // e.g. "BTC", "SOL"
  dir: "Long" | "Short",
  capital: number,        // USD capital
  entry: number,          // entry price
  sl: number,             // stop loss price
  tp: number,             // single take profit (OR use tpLevels)
  tpLevels: [             // partial takes (ventas parciales)
    { id, price, pct, hit }  // pct = % of position to close at this level
  ],
  capitalRemaining: number, // after partial takes
  be: boolean,            // breakeven flag
  date: string,
  note: string
}
```

### Direction Validation Rules
- **LONG**: SL < entry, TP > entry
- **SHORT**: SL > entry, TP < entry
- Always use `Math.abs()` for dollar amounts; validate sides separately

### Auto-Close Logic (`autoCheckPositions`)
Runs after every `fetchPrices()` cycle (~60s). Checks all open positions:
- SL hit вҶ’ closes position, saves to `xhist` with "рҹӣ‘ SL auto" or "вҡ–пёҸ BE auto" note
- TP level hit вҶ’ closes that % of position, marks level `hit:true`
- All TP levels hit вҶ’ removes position entirely, one consolidated xhist entry
- Sends Telegram + browser notification via `notifyAutoClose()`
- Saves immediately via `saveRef.current()` (no debounce) + synchronous localStorage write

### Ratio R:R Tracking
When a position closes (any method), `ratio` is saved in the `xhist` entry and accumulated in `ps.ratioSum` / `ps.ratioCount`. The Resumen tab shows "RATIO R:R MEDIO". **SL hits do NOT count toward ratio average вҖ” only TPs.**

### Manual Trade Add (Historial tab)
"+ AГұadir trade" button in Historial tab. Validates all fields and shows `alert()` on missing data (no silent failures). If close price is left empty, defaults to entry price (= breakeven). Writes to localStorage immediately on add.

## Alert System (AlertasTab)

### How Alerts Work
Each alert = one Binance WebSocket kline stream (`symbol@kline_interval`).
Historical candles pre-loaded via REST on start (250 candles).
OHLC stored in `ohlcRef.current[symbol+interval]`.
RSI history stored in `rsiHistRef.current[symbol+interval]`.
EMA state stored in `emaStateRef.current`.

### Automatic Triggers (no user toggle required)
- RSI enters вүӨ30 zone вҶ’ `"rsi_oversold"` (fires once on zone entry, **2-point hysteresis**: resets only when RSI > 32)
- RSI enters вүҘ70 zone вҶ’ `"rsi_overbought"` (resets only when RSI < 68)
- EMA 7 crosses above EMA 25 вҶ’ `"golden"`
- EMA 7 crosses below EMA 25 вҶ’ `"death"`
- EMA 50 crosses above EMA 200 вҶ’ `"ema200_golden"`
- EMA 50 crosses below EMA 200 вҶ’ `"ema200_death"`
- RSI divergence on closed candle вҶ’ `"rsi_div_bull"` or `"rsi_div_bear"` (time-based cooldown = 1 candle period)

**RSI and divergence are fully independent signals** вҖ” RSI notifications do NOT show divergence data.

### Optional User-Configured Triggers
- Custom RSI level (`rsiCustomEnabled`, `rsiCustomTarget`, `rsiCustomCondition`)
- FVG covered (`fvgEnabled`)

### Channel Detection (`detectChannelAlert`)
Detects ascending and descending parallel channels on closed candles only (`k.x === true`).

**Algorithm:**
1. Takes last 80 candles, detects pivot highs/lows with `detectPivots(highs, lows, 2)` (window=2 вҶ’ 4-candle confirmation)
2. `detectPivots` returns `{pH:[{x,y}], pL:[{x,y}]}` вҖ” **always use `.x`/`.y` accessors, never `[0]`/`[1]`**
3. Runs `linReg` on pivot arrays вҶ’ upper and lower trendlines
4. Parallelism check: same slope direction + ratio 0.20вҖ“5.0
5. Touch validation: pivot within 35% of channel height from its line counts as a touch (min 1 each side)
6. Breakout: price > topLineГ—1.002 (up) or price < botLineГ—0.998 (down)
7. False breakout: needs BOTH long wick AND close still inside channel
8. Zone alerts: `pos < 0.18` вҶ’ support, `pos > 0.82` вҶ’ resistance, middle вҶ’ no alert

**State machine (WS handler):**
- States: `null вҶ’ active вҶ’ break_pending вҶ’ confirmed вҶ’ retested вҶ’ done`
- `active`: channel detected, watching for breakout (close outside + вүҘ0.5 ATR displacement)
- `break_pending`: first candle closed outside вҖ” waits for second candle to confirm or reject
- `confirmed`: two consecutive closes outside вҶ’ fires `canal_{type}_ruptura_{dir}` alert
- `retested`: price returns to broken line (Вұ0.2 ATR) with candlestick confirmation вҶ’ fires retest alert
- `done`: retest fired or retest window (10Г— interval) expired
- Channel lost (`detectChannelAlert` returns null): resets all state to null

**Persistence (PR #32):** All `chan_*` fields are saved to `td-alert-zones` after every closed candle and restored in `startAlert()` with a staleness window of **3Г— interval**. If the saved state is older than 3 periods it is discarded and the machine starts fresh. This means breakout detection survives page reloads as long as the reload happens within 3 candle periods.

Channel state fields persisted per alert (key prefix `alertId_`):
`chan_type`, `chan_state`, `chan_break_dir`, `chan_break_line`, `chan_break_price`, `chan_confirm_ts`, `chan_atr_at_break`, `chan_height_at_break`, `chan_retest_done`, `chan_break_vol_ok`, `chan_last_updated`

Tests: `src/imbalances/chan-state-persist-tests.js` вҖ” 19 assertions. Run: `node src/imbalances/chan-state-persist-tests.js`

### Cooldown / Anti-spam
`lastTrigRef.current[key]` tracks fired state per alert+type. Fires once when condition becomes true, resets when it becomes false.
- RSI zones: 2-point hysteresis buffer (overbought resets at <68, oversold resets at >32)
- Divergence: time-based lock = one full candle period
- Channels: localStorage timestamp, cooldown = 3Г— interval period (survives WS reconnect)

### Alert Object Schema
```javascript
{
  id: number,
  symbol: string,           // e.g. "BTCUSDT"
  label: string,            // display name
  interval: "1h"|"4h"|"1d"|"1w",
  rsiCustomEnabled: boolean,
  rsiCustomTarget: number,  // 31-69
  rsiCustomCondition: "above"|"below",
  fvgEnabled: boolean,
  active: boolean,
  currentRsi: number|null,
  currentPrice: number|null,
  error: boolean
}
```

### Notifications
Every `sendAlert()` call:
1. Logs to in-app `logs` state (shown in Alertas tab)
2. Sets `lastAlert` state (banner shown for 8s)
3. Vibrates device (if supported)

5. Shows browser Notification (if permission granted)

## AuditorГӯa de operaciones вҖ” compatibilidad con registros histГіricos

El botГіn "Auditar" evalГәa operaciones cerradas contra 5 reglas de disciplina
(R1вҖ“R5, ver `AUDIT_SYSTEM_PROMPT`). Las operaciones cerradas antes de que
existiera la feature de auditorГӯa no tienen todos los campos enriquecidos
(`sl_initial`, `sl_modifications`, `thesis_text`, etc.). El auditor es
**tolerante**: audita con lo que hay y marca explГӯcitamente lo que no puede.

### ClasificaciГіn de campos

```javascript
AUDIT_CORE_FIELDS     = ["asset","dir","result","date","note"];
AUDIT_ENRICHED_FIELDS = ["sl_initial","sl_modifications","thesis_text",
                         "ratio","patternId","broker","reopen_after_sl"];
```

- **Core:** identidad de la operaciГіn. Si falta algo de esto вҶ’ la operaciГіn
  se marca `nonAuditable` (sin llamada a la API).
- **Enriched:** necesarios para puntuar reglas especГӯficas. Si faltan вҶ’ se
  informa al modelo en el prompt para que devuelva `status:"na"` en esas
  reglas, no `"fail"`.

### Helpers (top-level en `trading-diary.jsx`)

| FunciГіn | Rol |
|---|---|
| `detectMissingFields(op)` | `{core, enriched, hasMinimum}` |
| `hasMinimumAuditData(op)` | atajo booleano |
| `buildNonAuditableReport(op, missing)` | reporte sin AI para ops sin core |

### Flujo de `runAuditForEntry(entryId)`

1. Buscar `trade` en `D.current.xhist`. Si no existe вҶ’ `console.warn` y salir.
2. Calcular `missing = detectMissingFields(trade)`; `console.warn` si faltan.
3. Si `!missing.hasMinimum` вҶ’ persistir `nonAuditable` y volver.
4. Comprobar `localStorage.getItem("td-anthropic-key")`. Si estГЎ vacГӯo вҶ’
   `commitReport({error:"Configura tu API key de Anthropic en Ajustes вҶ’ AuditorГӯa."})` y volver.
5. Llamar `auditTrade(trade, null, missing)`; el prompt incluye los campos
   faltantes y pide marcarlos como `"na"`.
6. Sea cual sea el resultado (Г©xito, error API, excepciГіn), **siempre** se
   persiste un `audit_report`. Nunca se falla en silencio.

### Estados UI en el modal

| Estado | Trigger | Render |
|---|---|---|
| вң… pass | regla cumplida | badge verde |
| вқҢ fail | regla rota | badge rojo |
| рҹ”ҳ N/D | `status==="na"` | badge gris + explicaciГіn |
| AuditorГӯa limitada | `missingEnriched.length вүҘ 3` | badge `рҹ“Ӣ` en listado + nota en modal |
| No auditable | `nonAuditable:true` | panel ГЎmbar con razГіn + campos disponibles |
| Error API | `audit_report.error` | panel rojo + lista de campos faltantes + botГіn reaudit |

### Restricciones

- **No migrar datos histГіricos.** Las operaciones viejas conservan su
  estructura original. No se rellenan campos sintГ©ticamente.
- **No cambiar la lГіgica de las 5 reglas.** El auditor ya usa `"na"` para
  evaluaciones sin datos вҖ” solo hay que avisarle de quГ© falta.
- La racha `ps.tpStreak` y los KPIs (`avgScore`, `audited`, `pending`) filtran
  `nonAuditable` para no contaminar estadГӯsticas.

## Invariantes de direcciГіn en alertas FVG

Contratos que CUALQUIER cambio al mГіdulo de imbalances debe respetar. Auditado
en PR #23, tests en `src/imbalances/fvg-direction-tests.js`.

### I1 вҖ” ClasificaciГіn de direcciГіn
Los detectores (`checkFVGCovered`, `detectVolumeImbalance`, `detectOpeningGap`,
`detectLiquidityVoid`) DEBEN devolver objetos con:
- `subtype: "alcista" | "bajista"` вҖ” direcciГіn del imbalance
- `kind: "fvg" | "vi" | "gap" | "void"` вҖ” tipo
- `top`, `bot` вҖ” bordes numГ©ricos (top > bot siempre)

### I2 вҖ” Entrada en borde proximal
- **Long (FVG alcista) con precio arriba del FVG:** entry = `top` (primer borde
  que el precio toca al bajar).
- **Short (FVG bajista) con precio abajo del FVG:** entry = `bot` (primer borde
  que el precio toca al subir).

ImplementaciГіn correcta: `sendAlert` bloque `patron_fvg` lГӯnea ~4823
(`fvgEdge = fvgIsLong ? fvgR.top : fvgR.bot`).

### I3 вҖ” SL en el lado contrario + colchГіn ATR
- **Long:** SL `< bot` del FVG, restando `0.3 Г— ATR(14)` de la TF correspondiente.
- **Short:** SL `> top` del FVG, sumando `0.3 Г— ATR(14)`.

Nunca SL del mismo lado que la direcciГіn del trade.

### I4 вҖ” TP en la direcciГіn del trade
- **Long:** el siguiente imbalance usado como TP DEBE ser `subtype === "alcista"`
  y su precio `> entry`.
- **Short:** el siguiente imbalance usado como TP DEBE ser `subtype === "bajista"`
  y su precio `< entry`.

Prohibido usar un imbalance de direcciГіn contraria como TP (romperГӯa la lГіgica
operativa вҖ” el precio tendrГӯa que romper resistencia/soporte para llegar).

### I5 вҖ” `findFVGforTP` filtra por direcciГіn + posiciГіn
La funciГіn acepta `bullish: boolean` y filtra candidatos por:
1. DirecciГіn del imbalance (alcista si `bullish`, bajista si no).
2. PosiciГіn relativa a `refPrice` (por encima si bullish, por debajo si no).

`findNearFVG` filtra SOLO por direcciГіn y distancia вҖ” no por posiciГіn arriba/abajo
del `refPrice`. Usarla solo para detectar "dГіnde estamos parados", nunca para TP.

### I6 вҖ” R/R positivo y coherente (вүҘ 1.0)
- `R/R = |TP вҲ’ entry| / |entry вҲ’ SL|` usando `Math.abs()` вҶ’ siempre positivo.
- Si un cap de TP (por ejemplo `tpMaxPct` por TF en lГӯnea ~4661) recorta el TP
  de tal manera que `R/R < 1.0`, la seГұal DEBE marcarse "R/R insuficiente" o
  bloquearse вҖ” nunca emitirse como OPERACIГ“N vГЎlida.

### Tests
Cuatro casos canГіnicos en `src/imbalances/fvg-direction-tests.js`:

2. FVG bajista + precio arriba вҶ’ short con entry/SL/TP correctos.

## Cascada de TP por TF en multi-TF OPERACIГ“N

**Regla:** la TF **menor** de la confluencia dicta el rango vГЎlido de TP.

| TF menor de la confluencia | Rango TP vГЎlido |
|---|---|
| 1H | 0.8% вҖ“ 2.5% |
| 4H | 2.0% вҖ“ 5.0% |
| 1D | 4.0% вҖ“ 10.0% |
| 1W | 8.0% вҖ“ 25.0% |

**Ejemplo:** confluencia 1H + 1D вҶ’ `lowestTf = "1h"` вҶ’ TP debe estar en **0.8вҖ“2.5%** del entry.
Un FVG de 1D a +6% se rechaza aunque estГ© dentro del rango 1D, porque la TF menor (1H) manda.

### ImplementaciГіn

Variable `lowestTf` calculada justo despuГ©s de `tfConfs` (bloque multi-TF en `sendAlert`):
```javascript
var lowestTf = interval; // fallback
for (var i = 0; i < ["1h","4h","1d","1w"].length; i++) {
  if (tfConfs.some(c => c.tf === TF_LIST_ORD[i])) { lowestTf = TF_LIST_ORD[i]; break; }
}
```

`tpMaxPct` y `tpMinPct` se derivan de `TP_RANGES[lowestTf]` (no de `interval`).

2. FVG lejano multi-TF (`fvgFarExit`) dentro del mismo rango.
3. SeГұales especГӯficas de tipo (EMA50, canal opuesto, soporte/resistencia estructura).
4. **Swing estructural** de `ohlcByTf[lowestTf]` вҖ” Гәltimas 30 velas, nivel H/L mГЎs cercano en rango.
5. Si ninguno aplica вҶ’ `tpManualTarget = true` вҶ’ se emite la seГұal **sin TP** con nota `"gestiГіn manual del objetivo"`.

El cap final tambiГ©n respeta `tpMaxPct` derivado de `lowestTf`.

## PWA + Web Push Notifications

The app is an installable PWA on Android with native push notifications fanned
out by a Supabase Edge Function (`send-push`). Telegram runs in parallel as a
fallback during the transition window.

### Files

| File | Role |
|------|------|
| `manifest.json` | PWA manifest (install metadata, icons, shortcuts) |
| `sw.js` | Service Worker (push handler, notificationclick router, cache shell) |
| `icon.svg` | App icon вҖ” referenced by manifest and notifications |
| `migrations/add_push_subscriptions.sql` | `public.push_subscriptions` DDL (one row per device) |
| `supabase/functions/send-push/index.ts` | Edge Function: fan-out push to active subs via `web-push` |
| `scripts/generate-vapid.js` | Generate VAPID keypair (Node built-in crypto, no deps) |

### Setup (one-time)

```bash
# 1. Generate VAPID keys
node scripts/generate-vapid.js
# вҶ’ copy VAPID_PUBLIC_KEY into trading-diary.jsx (constant VAPID_PUBLIC_KEY)
# вҶ’ store both keys as Supabase secrets

# 2. Create the DB table
# Open Supabase SQL editor вҶ’ paste migrations/add_push_subscriptions.sql вҶ’ run

# 3. Set Edge Function secrets
supabase secrets set VAPID_PUBLIC_KEY=<public>
supabase secrets set VAPID_PRIVATE_KEY=<private>
supabase secrets set VAPID_SUBJECT=mailto:you@example.com

# 4. Deploy the Edge Function
supabase functions deploy send-push --no-verify-jwt

# 5. Rebuild and redeploy the frontend
node build.js && cp trading-diary.html index.html
git add вҖҰ && git commit -m "вҖҰ" && git push
```

### Subscription flow

1. User opens Alertas tab вҶ’ clicks рҹ”” button.
2. `subscribePush()` requests `Notification.permission` (Chrome Android shows OS prompt).
3. On grant: `pushManager.subscribe({userVisibleOnly:true, applicationServerKey:VAPID_PUBLIC_KEY})`.
4. Endpoint + `p256dh` + `auth` are POSTed to `push_subscriptions` via PostgREST.
5. Button turns green; the SW will now handle incoming push events even when the tab is closed.

Unsubscribe: pulsar el botГіn рҹ”” activo вҶ’ `sub.unsubscribe()` + PATCH row `is_active=false`.

### Payload format (client вҶ’ Edge Function)

```json
{
  "user_id": "miguel",
  "payload": {
    "title": "рҹҺҜ ORDEN LONG 4H вҖ” FVG alcista",

    "tag":   "td-BTC/USD-4h-patron_fvg",
    "icon":  "/icon.svg",
    "badge": "/icon.svg",
    "vibrate": [200, 100, 200],
    "data": {
      "deepLink":   "/?tab=7&sym=BTC%2FUSD&tf=4h",
      "signalType": "patron_fvg"
    },
    "actions": [
      {"action": "open_trade", "title": "AbrГӯ la orden"},
      {"action": "dismiss",    "title": "Ignorar"}
    ]
  }

```

### Deep link routing

The SW's `notificationclick` handler opens/focuses a client window at the
`deepLink` URL. `trading-diary.jsx` reads `?tab=N` on load and sets the active
tab. When `action=open_trade`, a flag is persisted to
`localStorage["td-pending-trade-action"]` so the Posiciones tab can pick it up.

Tab indices (see `TABS` array):

| Tab | Index | URL |
|---|---|---|
| Resumen | 0 | `/?tab=0` |
| Posiciones | 1 | `/?tab=1` |
| Historial | 2 | `/?tab=2` |
| Alertas | 7 | `/?tab=7` |
| Chat | 8 | `/?tab=8` |

### Fallback behaviour

- Telegram still fires on every `sendAlert` / `notifyAutoClose` вҶ’ push is purely additive right now.
- If the push endpoint returns 404/410 (expired subscription), the Edge Function marks it `is_active=false` automatically.
- `pushsubscriptionchange` event in the SW posts a message to open clients so the app can re-subscribe (automatic renewal: future enhancement).

### iOS note

Out of scope for this PR. iOS Safari requires the user to install the PWA to
the home screen **before** push works, and even then the VAPID delivery path
has quirks. Android Chrome is the primary target.
